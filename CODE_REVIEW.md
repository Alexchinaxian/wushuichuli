# IndustrialControlHMI 代码梳理计划

## 当前状态分析
- **总代码行数**: 8,309行
- **文件数量**: 53个文件 (C# + XAML)
- **架构**: MVVM模式，结构良好
- **技术栈**: .NET 8.0, WPF, Modbus, SQLite

## 发现的问题

### 1. 代码组织问题
- **转换器过多**: 13个转换器类，功能重复
- **模型分散**: 需要更好的分类和组织
- **服务层**: 接口和实现分离，但可以优化

### 2. 代码质量问题
- **空方法**: `FlowchartView_Loaded`方法为空
- **异常处理**: 需要更完善的错误处理
- **资源管理**: 需要检查Dispose实现

### 3. 配置管理
- 配置文件路径硬编码
- 缺少配置验证和默认值

### 4. 依赖注入
- 服务注册可以更模块化
- 缺少生命周期管理文档

## 改进计划

### 第一阶段：代码重组 (高优先级)

#### 1.1 合并转换器
**问题**: 13个转换器类，功能相似
**方案**: 
- 创建通用转换器基类
- 合并相似功能的转换器
- 使用参数化转换器

#### 1.2 重组模型层
**当前结构**:
```
Models/
├── AlarmRecord.cs
├── Enums.cs
├── KeyParameter.cs
├── NavigationItem.cs
├── ParameterItem.cs
├── Setting.cs
└── Flowchart/
    ├── CoordinatesConfig.cs
    ├── EquipmentItem.cs
    ├── FlowchartDataProvider.cs
    ├── FlowLineModel.cs
    ├── PlcPointMapping.cs
    ├── ProcessUnitModel.cs
    └── WiringConfigLoader.cs
```

**建议结构**:
```
Models/
├── Domain/           # 领域模型
│   ├── Alarm/
│   ├── Device/
│   ├── Process/
│   └── User/
├── ViewModels/       # 视图模型专用
├── DTOs/            # 数据传输对象
└── Enums/           # 枚举类型
```

#### 1.3 优化服务层
**当前**: 接口和实现在不同文件
**建议**: 使用文件夹组织
```
Services/
├── Modbus/
│   ├── IModbusService.cs
│   ├── ModbusService.cs
│   └── ModbusConfig.cs
├── Data/
├── Alarm/
└── Settings/
```

## 风险控制

### 高风险操作
1. **模型重组**: 可能影响数据绑定
2. **服务接口变更**: 影响依赖注入
3. **配置文件格式变更**: 影响部署

### 缓解措施
1. **逐步实施**: 小步快跑，频繁测试
2. **版本控制**: 使用Git分支管理
3. **备份**: 重要变更前创建备份
