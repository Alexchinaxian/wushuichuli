# 工艺流程图模块开发计划

## 1. 范围与目标

本计划针对上位机中的 **工艺流程图** 模块（FlowchartView + FlowchartRenderer，ProcessUnitControl / FlowLineControl，配置驱动布局与连接线），明确现状、缺口与分步实施任务。

**目标**：MBR 污水处理工艺流程在 WPF 上稳定展示、与 PLC 数据联动、支持交互与可维护配置。

**数据源约定**：
- **图源**：`参考资料/未命名绘图.drawio` 为工艺流程图的可视化源文件（draw.io 格式），画布尺寸、图元 ID、几何与连线均以该文件为准。
- **分析报告**：`参考资料/未命名绘图_分析报告.md` 由上述 drawio 解析生成，提供画布、设备、连接、工艺段、点位模板等结构化说明；配置与实现与报告一致即与 drawio 一致。
- **变更顺序**：修改流程图时先改 `未命名绘图.drawio`，再根据需要重新生成或人工同步《未命名绘图_分析报告》，最后更新 CoordinatesConfig 与代码。

**技术要求**：
1. **仅使用 C# 语言**：本模块及与本模块直接相关的工具、脚本、配置解析、drawio 解析等实现均须采用 C#（含 WPF/.NET）。
2. **禁止使用 Python**：不得引入或依赖 Python 脚本、Python 运行时或基于 Python 的辅助工具。
3. **使用相关技术复查界面可运行性及图形正确性**：须采用 C# 相关技术对工艺流程图界面进行复查，确认（1）界面可正常启动与运行，（2）流程图渲染结果与设计一致（与 `未命名绘图.drawio` / 《未命名绘图_分析报告》对照）。详见下文「5.1 界面与图形复查」。
4. **仅修改工艺部分的代码**：本计划范围内的开发与修改**仅限工艺流程图相关代码**（见下文「工艺部分范围」），不改动上位机其他模块（如通信、报警、历史、登录、主框架导航等）的实现；若必须调用或对接其他模块，仅通过既有接口或配置，不修改其内部代码。
5. **删除之前的无用代码**：在工艺部分范围内，须**清理并删除**无用、遗留、重复或已废弃的代码与配置（如与当前 drawio/分析报告 不一致的旧图元、未使用的类型或方法、冗余注释等），保持工艺流程图模块简洁、与设计一致。

**工艺部分范围**（允许修改的代码/配置）：
- 视图：FlowchartView、ProcessUnitControl、FlowLineControl 等流程图相关 XAML/代码隐藏。
- 视图模型与数据：FlowchartViewModel、FlowchartDataProvider、流程图用到的 Model（如 ProcessUnitModel、FlowLineModel）。
- 服务与渲染：FlowchartRenderer、流程图专用的绑定或转换逻辑。
- 配置与映射：CoordinatesConfig.json、流程图相关的 PlcPointMapping 或流程图专用配置；若 PlcDataBindingService 仅向流程图推送数据且无专用接口，其与流程图对接的调用方式可在工艺侧封装，但不改动通信/轮询核心逻辑（除非该服务归属工艺模块）。
- 测试与校验：仅针对工艺流程图模块的单元测试、配置校验工具、界面复查脚本（C#）。

## 2.3 如何使用《未命名绘图_分析报告》（详细说明）

开发工艺流程图模块时，**必须以** `参考资料/未命名绘图.drawio` 为图源、以 `参考资料/未命名绘图_分析报告.md` 为结构化数据源，与 `CoordinatesConfig.json`、`PlcPointMapping`、渲染与绑定逻辑一一对应。分析报告由 drawio 解析生成，若需核对原始几何或连线可直接打开 drawio。以下按报告章节说明用途与用法。

### 报告第一章：图元与画布信息

| 报告内容 | 用途 | 开发中的用法 |
|----------|------|-----------------------------|
| 图名（第 1 页） | 确认图纸版本 | 配置或注释中可标注来源页 |
| **pageWidth=827, pageHeight=1169** | 画布尺寸 | FlowchartView/画布宽高应与此一致或按比例缩放，保证与 drawio 导出一致 |
| **gridSize=10** | 网格步长 | 对齐、吸附、坐标取整时可参考（如坐标按 10 的倍数） |

→ **第一阶段 1.2「画布尺寸与缩放」**：直接采用报告中的 pageWidth/pageHeight 作为画布基准尺寸，再根据主内容区做缩放或滚动。

---

### 报告第二章：设备框明细

| 报告小节 | 内容 | 开发中的用法 |
|----------|------|-----------------------------|
| **2.1 设备总表** | 每行：图元ID、设备名称、左上角(x,y)、右下角(x,y)、宽×高、**中心(x,y)** | **CoordinatesConfig 的 Items**：`ElementId` = 报告中的「图元ID」；`Name`/显示名 =「设备名称」；位置建议用**中心坐标** (centerX, centerY)，或由左上角+宽高推算；类型（池体/泵/阀门等）可来自 2.2 |
| **2.2 按设备类型分类** | 构筑物、泵类、风机/搅拌、阀门、补水、加药、除臭、其他 | 用于 ProcessUnitControl 的样式/图标、图例分类、以及 PlcPointMapping 中按类型分组维护 |
| **2.3 同名设备区分** | 同名设备（如两处「自来水补水」「电磁阀」「加药装置」）用位置或图元ID区分 | **必须用图元ID 作为唯一键**，不能仅用名称；UI 显示名可加后缀（如「自来水补水(左)」）以与报告一致 |

→ **第一阶段 1.1「核对 CoordinatesConfig 与 drawio 分析」**：逐行对照报告 2.1 表格，确保 32 个 Items 的 ElementId、名称、坐标、类型与报告一致；同名设备用报告 2.3 的区分方式在配置与界面中统一。

---

### 报告第三章：连接关系明细

| 报告小节 | 内容 | 开发中的用法 |
|----------|------|-----------------------------|
| **3.1 有明确 source→target 的边** | 边ID、源设备（图元ID）、目标设备（图元ID）、源→目标名称 | **CoordinatesConfig 的 FlowLines**：每条线 `SourceId`/`TargetId` 取报告中的「源设备（图元ID）」「目标设备（图元ID）」；线型/颜色可结合 4.x 工艺段再细分为 MainProcess/WaterSupply/Dosing/Reflux/Backwash 等 |
| **3.2 连线拐点** | 含折线路径的边及其拐点坐标序列 | FlowLines 中若支持 **Waypoints/Points**，则按报告「拐点坐标（按顺序）」填入，实现与 drawio 一致的折线走向 |
| **3.3 终点为 targetPoint 的边** | 源设备、targetPoint 坐标、推断目标 | 若当前配置只支持「设备→设备」，可先将 targetPoint 转为最邻近目标设备或虚拟节点；若支持「设备→坐标」，则直接使用报告中的 targetPoint |

→ **第一阶段 1.1**：27 条（及以上）FlowLines 的 Source/Target 与报告 3.1 一一对应；带拐点的线用 3.2 补全路径。**第一阶段 1.3「连接线类型与样式」**：可结合报告第四章工艺段给每条线打类型标签。

---

### 报告第四章：按工艺段的连接汇总

| 报告小节 | 内容 | 开发中的用法 |
|----------|------|-----------------------------|
| **4.1～4.5** | 进水与预处理→调节池、调节池→缺氧池、缺氧池↔MBR、MBR→产水与反洗、中间水池→回用与补水/加药 | 用于给 **FlowLines 打 LineType**（如 MainProcess / WaterSupply / Dosing / Reflux / Backwash），便于 1.3 按类型设颜色与线型；也用于图例说明与文档 |

---

### 报告第五章：未与其他设备连线的图元

| 报告内容 | 开发中的用法 |
|----------|-----------------------------|
| 搅拌机、碳源投加、缺氧气搅拌机等「仅位置、无连接」 | 这些图元仍要在 **CoordinatesConfig 的 Items** 中配置，保证在画布上显示；FlowLines 不为其建线。PlcPointMapping 中若需显示状态，照常按 UnitId/图元ID 映射 |

---

### 报告第六章：入口/出口锚点摘要

| 报告内容 | 开发中的用法 |
|----------|-----------------------------|
| 源 exitX,exitY、目标 entryX,entryY（归一化 0～1） | 若 FlowLine 渲染支持「从设备某边出、进目标某边」，可把报告中对应连接的出口/入口锚点写入配置，使连线更贴近原图 |

---

### 报告第七、八章：管线编号与 SCADA 点位

| 报告章节 | 内容 | 开发中的用法 |
|----------|------|-----------------------------|
| **第七章** | 管线编号/位号空白模板（源→目标、介质、管径等） | 可选：用于图例、工具提示或报表中的管线描述 |
| **第八章** | 按设备/工艺的 SCADA 点位空白模板（泵类、风机与搅拌、阀门、池体液位、加药等） | **PlcPointMapping** 与 **PlcDataBindingService** 填表时，直接按 8.1～8.5 的设备名称与建议位号，与 PLC 变量表对照填写；便于第二阶段 2.1「完善 PlcPointMapping 与流程图对应」 |

---

### 小结：报告与开发阶段的对应关系

| 开发阶段/任务 | 主要使用的报告章节 |
|----------------|---------------------|
| 1.1 配置与 drawio 一致 | 第二章 2.1/2.2/2.3，第三章 3.1/3.2/3.3 |
| 1.2 画布尺寸与缩放 | 第一章 pageWidth/pageHeight/gridSize |
| 1.3 连接线类型与样式 | 第三章 3.1，第四章 4.1～4.5 |
| 1.4 图例与说明 | 第二章 2.2（类型），第四章（工艺段） |
| 2.1 点位映射与流程图对应 | 第二章 2.1（设备名与图元ID），第八章 8.1～8.5 |
| 2.2～2.4 状态与数据绑定 | 第八章设备列表 + 现有 PlcPointMapping 结构 |
| 4.2 点位映射配置化 / 4.3 增删约定 | 第八章模板 + 第二章 2.3（同名区分） |

**规范**：凡修改设备数量、连接关系、画布尺寸或管线类型时，应**先修改** `参考资料/未命名绘图.drawio`，再同步《未命名绘图_分析报告》（必要时用解析脚本重新生成），最后改 CoordinatesConfig 与代码，保证图源 → 报告 → 配置与实现一致。

## 5. 风险与注意

### 5.1 界面与图形复查（相关技术）

须使用 **C# 相关技术** 对工艺流程图模块进行复查，确保**界面可运行**且**图形正确**。以下方式可单独或组合使用（均须在 C#/.NET 范围内实现）：

| 复查项 | 说明 | 可选技术/手段 |
|--------|------|-------------------------------|
| **界面可运行** | 流程图视图能正常打开、不崩溃、无阻塞；画布与控件可交互（滚动/缩放/点击等） | C# 单元测试（ViewModel/DataProvider 等）、集成测试（启动主窗并导航至流程图）、WPF UI Automation（UIA）自动化打开流程图页并做简单操作 |
| **图形正确** | 设备数量、位置、连线关系及拐点与《未命名绘图_分析报告》及 drawio 一致；线型/颜色符合工艺段定义 | 基于报告与 CoordinatesConfig 的**自动化校验**：用 C# 读取配置与报告数据，比对 Item 数量、ElementId 集合、FlowLines 的 Source/Target/拐点；可选：渲染后通过 UIA 或截图与参考图对比（像素/特征比对或人工抽查） |
| **回归与持续** | 配置或 drawio 变更后，能快速确认界面与图形未退化 | 将上述校验纳入 C# 测试工程（如 xUnit/NUnit），在 CI 或本地构建中运行；关键路径可配合少量 UI 自动化用例 |

**实施建议**：
- 优先做**配置与报告一致性校验**（C# 控制台或测试项目）：加载 CoordinatesConfig.json，对照分析报告中的设备表、连接表做 ElementId、数量、Source/Target 等断言。
- 再补**界面可运行性**：集成测试或 UIA 脚本（C#）启动应用并打开流程图视图，断言窗口存在、无未处理异常。
- 图形正确性中的**布局/坐标**可通过“配置已与报告一致”间接保证；若需显式验证渲染结果，可采用 C# 的截图 + 简单比对或人工复查清单。

---

### 5.2 其他风险与注意

| 项 | 说明 |
|----|------|
| ElementId 与 PLC 命名不一致 | 在 PlcPointMapping 中做显式映射，避免强依赖命名一致 |
| 设备数量或连接关系变更 | 以 `未命名绘图.drawio` 为图源；变更时先改 drawio，再同步分析报告与 CoordinatesConfig |
| 性能 | 单元/连接线数量可控时 WPF 足够；若后续数量大增，再考虑虚拟化或简化绘制 |
